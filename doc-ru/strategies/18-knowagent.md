# Стратегия KnowAgent

**Модуль:** `intent-reactor-strategies`
**Значение стратегии:** `knowagent`

---

## Краткое описание

KnowAgent — декоратор над базовым планировщиком ReACT, обогащающий планирование Базой Знаний о Действиях (KB). KB фиксирует предусловия, постусловия и противопоказания для каждого доступного инструмента — извлечённые из описаний инструментов и при необходимости дополненные вызовом LLM. Перед каждой итерацией планирования инструменты, предусловия которых не выполнены в текущем состоянии сессии, исключаются из списка доступных — это снижает риск вызова инструментов в неправильном порядке или без нужного контекста.

---

## Принцип работы

**Шаг 1 — Инициализация KB** *(один раз за сессию)*
При первом вызове KnowAgent строит KB из метаданных инструментов:

- **Извлечение по паттернам** — регулярные выражения сканируют описание каждого инструмента на языковые маркеры предусловий (`requires`, `must first`, `after calling`…), постусловий (`sets`, `creates`, `returns`…) и противопоказаний (`do not`, `never`, `avoid`…).
- **LLM-обогащение** *(если `enrich-knowledge: true`)* — единый промпт обрабатывает все инструменты и возвращает структурированный JSON с более детальными предусловиями, постусловиями и противопоказаниями. Добавляет один вызов LLM при старте сессии.

KB сохраняется в `session.attributes["knowagent_kb"]` и переиспользуется во всех последующих вызовах.

**Шаг 2 — Фильтрация инструментов** *(каждый вызов, если `filter-by-preconditions: true`)*
Перед делегированием в базовый ReACT-планировщик KnowAgent анализирует историю сообщений сессии на предмет наличия свидетельств выполнения предусловий каждого инструмента (например, сообщение `[TOOL_RESULT]` от инструмента-предшественника). Инструменты с неудовлетворёнными предусловиями удаляются из списка доступных для данного шага планирования.

**Шаг 3 — Инжекция контекста KB**
KB форматируется в виде системного сообщения и добавляется к сообщениям, передаваемым базовому ReACT-планировщику — чтобы LLM был осведомлён о связях между инструментами даже для тех, что не были отфильтрованы.

**Шаг 4 — Делегирование**
Планирование полностью делегируется `DefaultReACTPlanner`. KnowAgent лишь оборачивает список инструментов и контекст сообщений.

---

## Атрибуты сессии

| Атрибут | Описание |
|---|---|
| `knowagent_kb` | `Map<String, ToolKnowledge>` — предусловия, постусловия, противопоказания по каждому инструменту |
| `knowagent_initialized` | `Boolean` — была ли KB построена для данной сессии |

---

## Конфигурация

```yaml
intent-reactor:
  planning:
    strategy: knowagent
    strategies:
      know-agent:
        enrich-knowledge: false    # true = LLM-обогащение KB при первом вызове (доп. задержка)
        filter-by-preconditions: true # исключать инструменты с неудовлетворёнными предусловиями
    max-steps: 10
```

Установите `enrich-knowledge: true` для сценариев с большим числом инструментов, имеющих сложные взаимозависимости, не отражённые полностью в их описаниях. Вызов LLM для обогащения выполняется один раз за сессию.

---

## Когда использовать

- Сценарии с множеством инструментов, имеющих явные требования к порядку вызова.
- Ситуации, когда LLM систематически вызывает инструменты не по порядку или без нужного контекста.
- Любой ReACT-сценарий, где фильтрация по предусловиям может снизить количество ошибочных вызовов инструментов.

---

## Компромиссы

| Плюсы | Минусы |
|---|---|
| Снижает количество некорректных вызовов инструментов | `enrich-knowledge` добавляет разовый вызов LLM |
| KB-контекст улучшает понимание LLM связей между инструментами | Извлечение по паттернам может пропустить неявные предусловия |
| Полная совместимость со всеми возможностями ReACT | Фильтрация может исключить валидные инструменты при неточном определении предусловий |
| Нет изменений в коде — замена `react` без модификаций | Небольшие накладные расходы на каждый вызов для фильтрации и инжекции контекста |
