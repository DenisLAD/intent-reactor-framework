# Жизненный цикл запроса

Полное описание того, что происходит при вызове `IntentReactorService.process(sessionId, message)`.

---

## Пошаговый поток

### 1. Загрузка или создание сессии

`SessionStore.load(sessionId)` вызывается первым. Если сессия не найдена — создаётся новый `SessionState` с пустой историей сообщений и пустыми атрибутами.

### 2. Закрепление сообщения

Входящее сообщение пользователя добавляется в `session.messages`. Сообщение автоматически закрепляется в двух случаях:
- Это первое сообщение в сессии (исходная цель никогда не теряется).
- В атрибутах сессии установлен флаг `PIN_NEXT_USER_MESSAGE = true`.

Закреплённые сообщения не вытесняются скользящим окном контекста и не подвергаются компрессии.

### 3. Анализ намерений

```
IntentPreprocessor.analyze(message, session) → IntentAnalysisResult
```

LLM классифицирует сообщение на одно или несколько намерений. Каждое намерение содержит имя, описание и уровень уверенности. Публикуются события `IntentAnalysisStartedEvent` и `IntentAnalysisCompletedEvent`.

### 4. Инициализация состояния планирования

Создаётся `PlanState` со счётчиком шагов `stepCount = 0` и статусом `IN_PROGRESS`. Если у результата несколько намерений — активируется логика мультинамерений (шаг 5). В противном случае сразу начинается цикл ReACT (шаг 6).

### 5. Диспетчеризация мультинамерений (если намерений > 1)

В зависимости от `planning.multi-intent.strategy`:

| Стратегия | Поведение |
|---|---|
| `sequential` | Намерения обрабатываются по очереди, каждое проходит полный ReACT-цикл |
| `parallel` | Для каждого намерения клонируется сессия; циклы выполняются в `CompletableFuture` |
| `llm-driven` | LLM упорядочивает намерения; при ошибке — сортировка по уровню уверенности |

Подробнее — в [Мультинамерениях](06-multi-intent.md).

### 6. Цикл ReACT

Планировщик вызывается в цикле. Каждая итерация:

#### 6a. Возобновление после подтверждения

Если в `session.attributes["pendingStep"]` сохранён ожидающий шаг — он обрабатывается немедленно. Этот путь используется при возобновлении через `proceedAfterConfirmation()`.

#### 6b. Вызов планировщика

```
Planner.plan(session, intent) → Plan
```

Для `DefaultReACTPlanner` / `ReflexionPlanner` перед отправкой истории в LLM выполняются два конвейера расширений (подробнее — в [Управлении контекстным окном](08-context-window.md)):
1. **Цепочка `MessageContextPreProcessor`** — на полном списке сообщений сессии (до скользящего окна)
2. Скользящее окно + повторная вставка закреплённых сообщений
3. **Цепочка `MessageContextPostProcessor`** — на сообщениях в окне (компрессия, дедупликация и т. д.)

`LATSPlanner` эти конвейеры не использует.

#### 6c. Обработка типа шага

**`REASON`**
- Рассуждение добавляется в атрибуты сессии в виде `ReasoningStep`.
- Не добавляется в `session.messages` (LLM не видит собственные рассуждения напрямую).
- Цикл продолжается.

**`ACT`**
- Если `tool.isRisky() && !autonomous` и `ConfirmationManager.needsConfirmation()` возвращает `true`:
  - Шаг сохраняется в `session.attributes["pendingStep"]`.
  - Публикуется `ConfirmationRequiredEvent`.
  - Возвращается `ReactorResponse` со статусом `AWAITING_CONFIRMATION`.
  - Вызывающий код должен дождаться ответа пользователя.
- В противном случае инструмент выполняется немедленно.
- Результат добавляется в `session.messages` как сообщение с ролью `SYSTEM`.
- Публикуется `PlanStepCompletedEvent`.
- Цикл продолжается.

**`DONE`**
- `output` становится финальным текстом.
- Публикуется `PlanCompletedEvent`.
- Цикл завершается.

**`FAIL`**
- `output` содержит причину неудачи.
- Публикуется `PlanFailedEvent`.
- Цикл завершается.

#### 6d. Превышение лимита шагов

Если `planState.stepCount >= planning.max-steps`:
- Публикуется `PlanFailedEvent` с причиной `"max steps exceeded"`.
- Возвращается соответствующий `ReactorResponse`.

### 7. Сборка ответа

`ReactorResponse` содержит:

| Поле | Описание |
|---|---|
| `text` | Финальный текст ответа из шага `DONE` |
| `status` | `COMPLETED` / `FAILED` / `AWAITING_CONFIRMATION` |
| `performedActions` | Список всех выполненных вызовов инструментов с результатами |
| `reasoningSteps` | Все шаги `REASON`, собранные в ходе планирования |
| `sessionId` | Идентификатор сессии для следующего вызова |

## Диаграмма последовательности

```
Клиент          Service           Preprocessor     Planner       Tool
  │                │                    │              │             │
  │──process()────▶│                    │              │             │
  │                │──analyze()────────▶│              │             │
  │                │◀──IntentResult─────│              │             │
  │                │                    │              │             │
  │                │──────────────────────plan()──────▶│             │
  │                │◀─────────────────────PlanStep─────│             │
  │                │                    │              │             │
  │                │  [если ACT]        │              │             │
  │                │──────────────────────────execute()────────────▶│
  │                │◀─────────────────────────────ToolResult─────────│
  │                │                    │              │             │
  │                │──────────────────────plan()──────▶│  (следующая
  │                │◀─────────────────────DONE─────────│   итерация)
  │                │                    │              │             │
  │◀──ReactorResponse──────────────────│              │             │
```
