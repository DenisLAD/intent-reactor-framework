# Стратегия MAP (Modular Agentic Planner)

**Модуль:** `intent-reactor-strategies`
**Значение стратегии:** `map`

---

## Краткое описание

MAP (Модульный Агентный Планировщик) оркестрирует пять специализированных когнитивных модулей LLM, каждый из которых отвечает за отдельный аспект процесса планирования. Вместо единого монолитного планировщика MAP делегирует декомпозицию, предсказание состояния, оценку прогресса, обнаружение конфликтов и координацию действий отдельным промптам, вызываемым в разные моменты жизненного цикла выполнения.

---

## Принцип работы

**Фаза 1 — PLAN** *(один вызов, до любого исполнения инструментов)*
Четыре LLM-модуля выполняются последовательно для формирования плана с высокой уверенностью:

1. **TaskDecomposer** — декомпозирует общую цель в до `max-subtasks` упорядоченных подзадач. Каждая подзадача — конкретный, адресуемый инструментами шаг с опциональными ссылками на зависимости.
2. **StatePredictor** *(если `use-state-predictor: true`)* — для каждой подзадачи предсказывает ожидаемое промежуточное состояние и инжектирует предсказания как SYSTEM-сообщения.
3. **Evaluator** — оценивает предложенный план по шкале 0.0–1.0. Если оценка достигает `confidence-threshold` и конфликтов нет — план принят.
4. **ConflictMonitor** *(если `use-conflict-monitor: true`)* — проверяет подзадачи на ресурсные или порядковые конфликты. При обнаружении конфликта его описание передаётся в следующий вызов TaskDecomposer.

Шаги 2–4 повторяются до `max-planning-iterations` раз. После принятия плана он инжектируется в сессию как закреплённое пользовательское сообщение.

**Фаза 2 — EXECUTE**
Все последующие вызовы `plan()` полностью делегируются базовому `DefaultReACTPlanner`, который пошагово исполняет список подзадач. Сам MAP в фазе исполнения не участвует.

---

## Атрибуты сессии

| Атрибут | Описание |
|---|---|
| `map_phase` | Текущая фаза: `PLAN` / `EXECUTE` |

Всё состояние модулей (список подзадач, оценки, контекст конфликтов) локально для одного вызова `plan()`. После завершения фазы PLAN финальный план инжектируется как закреплённое пользовательское сообщение, и сессия переходит в EXECUTE — все последующие вызовы делегируются базовому планировщику ReACT.

---

## Конфигурация

```yaml
intent-reactor:
  planning:
    strategy: map
    strategies:
      map:
        max-subtasks: 5              # максимальное число подзадач от TaskDecomposer
        max-planning-iterations: 3   # итерации уточнения (декомпозиция → оценка → мониторинг)
        confidence-threshold: 0.7    # оценка Evaluator, необходимая для принятия плана
        use-conflict-monitor: true   # запускать ConflictMonitor после каждой оценки
        use-state-predictor: false   # StatePredictor для каждой подзадачи (доп. вызов LLM)
    max-steps: 50
```

---

## Когда использовать

- Задачи, требующие структурированного контроля прогресса, а не открытой итерации.
- Сценарии, где ошибки инструментов часто вызваны противоречиями состояния или неудовлетворёнными предусловиями.
- Многошаговые процессы с чёткой декомпозицией на подзадачи.

---

## Компромиссы

| Плюсы | Минусы |
|---|---|
| Структурированная декомпозиция снижает дрейф | Больше вызовов LLM, чем в чистом ReACT |
| Evaluator предотвращает избыточное выполнение | ConflictMonitor/StatePredictor добавляют задержку |
| ConflictMonitor восстанавливается после противоречий | Качество декомпозиции зависит от возможностей LLM |
| Модульность — каждый модуль можно включить/выключить | Сложнее анализировать, чем линейные планировщики |
