# Стратегия MAP (Modular Agentic Planner)

**Модуль:** `intent-reactor-strategies`
**Значение стратегии:** `map`

---

## Краткое описание

MAP (Модульный Агентный Планировщик) оркестрирует пять специализированных когнитивных модулей LLM, каждый из которых отвечает за отдельный аспект процесса планирования. Вместо единого монолитного планировщика MAP делегирует декомпозицию, предсказание состояния, оценку прогресса, обнаружение конфликтов и координацию действий отдельным промптам, вызываемым в разные моменты жизненного цикла выполнения.

---

## Принцип работы

**Модуль 1 — TaskDecomposer** *(один раз, при первом вызове)*
Декомпозирует общую цель в упорядоченный список атомарных подзадач. Каждая подзадача — конкретный, адресуемый инструментами шаг.

**Модуль 2 — TaskCoordinator** *(каждый шаг)*
Исходя из текущей подзадачи, доступных инструментов и последних наблюдений, решает следующее действие: какой инструмент вызвать (или шаг REASON) и с какими параметрами.

**Модуль 3 — Evaluator** *(каждые `eval-interval-steps` шагов)*
Оценивает общий прогресс по шкале 0.0–1.0. При превышении `progress-threshold` планировщик переходит к синтезу — это предотвращает лишние шаги после фактического достижения цели.

**Модуль 4 — ConflictMonitor** *(при ошибках инструментов, если `use-conflict-monitor: true`)*
При неудаче вызова инструмента определяет характер ошибки: противоречие (конфликтующие предположения о состоянии), неудовлетворённое предусловие или временная ошибка. Диагноз инжектируется в следующий вызов TaskCoordinator.

**Модуль 5 — StatePredictor** *(каждый шаг, если `use-state-predictor: true`)*
Перед исполнением действия предсказывает ожидаемый результат. Предсказание сравнивается с фактическим наблюдением, а дельта передаётся обратно в TaskCoordinator. По умолчанию отключён из-за дополнительного вызова LLM на каждый шаг.

---

## Атрибуты сессии

| Атрибут | Описание |
|---|---|
| `map_phase` | Текущая фаза: `DECOMPOSE` / `COORDINATE` / `SYNTHESIZE` |
| `map_subtasks` | `List<String>` — список декомпозированных подзадач |
| `map_subtask_index` | Индекс текущей активной подзадачи |
| `map_eval_score` | Последняя оценка Evaluator (0.0–1.0) |
| `map_eval_step_count` | Шагов с последнего запуска Evaluator |
| `map_conflict_context` | Диагноз ConflictMonitor после последней ошибки |

---

## Конфигурация

```yaml
intent-reactor:
  planning:
    strategy: map
    strategies:
      map:
        max-subtasks: 5           # максимальное число подзадач от TaskDecomposer
        progress-threshold: 0.8   # оценка Evaluator для перехода к синтезу
        use-conflict-monitor: true # ConflictMonitor при ошибках инструментов
        use-state-predictor: false # StatePredictor (доп. вызов LLM на шаг)
        eval-interval-steps: 3    # запускать Evaluator каждые N шагов
    max-steps: 50
```

---

## Когда использовать

- Задачи, требующие структурированного контроля прогресса, а не открытой итерации.
- Сценарии, где ошибки инструментов часто вызваны противоречиями состояния или неудовлетворёнными предусловиями.
- Многошаговые процессы с чёткой декомпозицией на подзадачи.

---

## Компромиссы

| Плюсы | Минусы |
|---|---|
| Структурированная декомпозиция снижает дрейф | Больше вызовов LLM, чем в чистом ReACT |
| Evaluator предотвращает избыточное выполнение | ConflictMonitor/StatePredictor добавляют задержку |
| ConflictMonitor восстанавливается после противоречий | Качество декомпозиции зависит от возможностей LLM |
| Модульность — каждый модуль можно включить/выключить | Сложнее анализировать, чем линейные планировщики |
