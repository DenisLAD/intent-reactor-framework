# Стратегия Plan-and-Solve

**Модуль:** `intent-reactor-strategies`
**Значение стратегии:** `plan-and-solve`

---

## Краткое описание

Plan-and-Solve разделяет планирование и выполнение на две явные фазы. Сначала LLM составляет полный нумерованный план для задачи. Затем план выполняется шаг за шагом, причём каждый шаг может использовать инструменты. Это снижает арифметические и логические ошибки, заставляя LLM думать наперёд перед действием.

---

## Принцип работы

1. **Фаза планирования** — LLM получает цель и его просят разработать полный исполняемый план с нумерованными шагами: *«Сначала составь план. Затем выполни его.»* План — список конкретных действий.
2. **Фаза выполнения** — каждый шаг плана последовательно передаётся циклу ReACT. Цикл ReACT может вызывать инструменты для каждого шага. Результат каждого шага сохраняется.
3. После завершения всех шагов (или при выдаче шага `FAIL`) финальный LLM-вызов синтеза собирает результаты шагов в связный ответ.
4. Выдаётся шаг `DONE`.

---

## Конфигурация

```yaml
intent-reactor:
  planning:
    strategy: plan-and-solve
    max-steps: 50   # применяется к фазе выполнения
```

## Промпты

| Промпт | Путь в classpath | Назначение |
|---|---|---|
| Промпт планирования | `prompts/ps-plan.md` | Генерирует нумерованный план выполнения |
| Промпт выполнения шага | `prompts/react-system.md` | Цикл ReACT для каждого шага |
| Промпт синтеза | `prompts/ps-synthesize.md` | Объединяет выводы шагов в финальный ответ |

---

## Когда использовать

- Многошаговые задачи с чётким началом и концом: *«Проанализируй кодовую базу, выяви проблемы и сгенерируй отчёт о рефакторинге.»*
- Задачи, где письменный план помогает выявить пропущенные шаги до выполнения.
- Когда нужно чётко разделить фазы планирования и выполнения для аудита.
- Структурированные рабочие процессы автоматизации.

---

## Сравнение плюсов и минусов

| Преимущества | Недостатки |
|---|---|
| Явный план инспектируемый и логируемый | Жёсткость — план может плохо адаптироваться к неожиданным результатам |
| Снижает путаницу при многошаговом выполнении | Дополнительный LLM-вызов для планирования |
| Ниже частота ошибок при многошаговой арифметике/логике | Избыточное планирование для простых задач |

---

## Отличие от ReACT

| | `react` | `plan-and-solve` |
|---|---|---|
| Создание плана | Неявное, по одному шагу | Явный полный план заранее |
| Адаптивность | Высокая — может изменить курс на каждом шаге | Ниже — следует начальному плану |
| Аудируемость | Через историю шагов | Через явный документ плана |

---

## Пример

```yaml
intent-reactor:
  planning:
    strategy: plan-and-solve
```

Цель: *«Суммируй все .log-файлы в /var/logs и подсчитай общее количество строк с ошибками.»*

Сгенерированный план:
1. Найти все .log-файлы в /var/logs.
2. Прочитать каждый файл.
3. Подсчитать строки, содержащие «ERROR».
4. Резюмировать: всего файлов, всего ошибок, наиболее частая ошибка.

Выполнение: цикл ReACT вызывает `glob`, `read_file`, `calculator` согласно плану, затем синтезирует резюме.
